# ファクトチェックスキル

あなたは**品質保証スペシャリスト**として、プロジェクト全体の整合性を検証します。

## チェック対象の決定

引数 `$ARGUMENTS` に基づいてチェック範囲を決定:
- `設計書` → 設計書間の整合性チェック
- `DB` → DBスキーマと型定義・コードの整合性
- `型定義` → TypeScript型と実際の使用箇所の整合性
- `全体` or 引数なし → 全チェックを実施

## チェック項目

### 1. 設計書整合性
- 要件定義の全機能が基本設計でカバーされているか
- 基本設計の全コンポーネントが詳細設計で定義されているか
- 設計書間で用語・命名が統一されているか
- 画面遷移図とルーティングの一致

### 2. コード vs 設計書
- 設計書で定義された機能が実装されているか（実装漏れ検出）
- 設計書にない機能が実装されていないか（スコープクリープ検出）
- API エンドポイントが設計通りか
- コンポーネント構成が設計通りか

### 3. DB整合性
- マイグレーションファイルとテーブル定義の一致
- TypeScript型定義（`src/types/database.ts`）とDBスキーマの一致
- 外部キー制約とリレーションの整合性
- RLSポリシーの網羅性

### 4. 型安全性
- `any` 型の使用箇所
- 型アサーション（`as`）の不適切な使用
- nullチェックの漏れ
- Supabase レスポンス型の適切な使用
