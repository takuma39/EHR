# 整体院向けスマートカルテ 要件定義書

## 1. プロジェクト概要

- **コンセプト**: 患者が自らタブレットで入力する、AI親和性の高い次世代型電子カルテUI。
- **ターゲット**: 整体院・接骨院のスタッフ（管理者）および来院患者。
- **デザイン方針**: 清潔感と科学的信頼感のあるデザイン。
- **想定規模**: タブレット1台、登録患者数百人程度の小規模院。

## 2. システム要件 / 技術スタック

- **Frontend**: React (Next.js 14+ App Router), TypeScript (strict mode)
- **Styling**: Tailwind CSS + shadcn/ui
- **Animation**: Framer Motion (スムーズな画面遷移用)
- **Backend/DB**: Supabase (Auth, Database, RLS)
- **Deployment**: Vercel

## 3. ユーザーストーリー

### 患者として

- [x] **US-P01** [Must]: 患者として、タブレットで「受付を始める」ボタンを押して入力を開始したい。なぜなら、スタッフを待たずに自分で手続きを進められるから。
- [x] **US-P02** [Must]: 患者として、氏名・生年月日・電話番号・住所を入力したい。なぜなら、来院記録を自分の情報と紐付ける必要があるから。
- [x] **US-P03** [Must]: 患者として、人体図から痛みのある部位をタップで選択したい。なぜなら、文字で説明するより直感的に伝えられるから。
- [x] **US-P04** [Must]: 患者として、痛みの強さをスライダーで入力したい。なぜなら、段階的な強さを簡単に伝えられるから。
- [x] **US-P05** [Must]: 患者として、発症時期を選択肢から選びたい。なぜなら、いつから症状があるかを正確に伝えることで施術の参考になるから。
- [x] **US-P06** [Should]: 患者として、自由記述欄に補足を入力したい。なぜなら、部位選択だけでは伝えきれない詳細があるから。
- [x] **US-P07** [Must]: 患者として、入力完了後に完了メッセージを見たい。なぜなら、入力が正しく送信されたことを確認したいから。
- [x] **US-P08** [Must]: 患者として、HPから初回の来院予約をしたい。なぜなら、電話しなくても好きな時間に予約できるから。
- [x] **US-P09** [Must]: 患者として、HP予約完了後に確認メールを受け取りたい。なぜなら、予約日時を手元に残しておきたいから。
- [x] **US-P10** [Must]: 患者として、施術後に店内で次回予約をしたい。なぜなら、店舗側が次回の予約をその場で確実に確保（FIX）したいから。（※患者側の意向というより、店舗側の再来店率向上のための業務要件）
- [x] **US-P11** [Should]: 患者として、HPから次回の来院予約をしたい。なぜなら、自宅から都合の良い日時を選びたいから。
- [x] **US-P13** [Must]: 患者として、マイページにログインしたい。なぜなら、自分の予約状況や登録情報を管理したいから。
- [x] **US-P14** [Must]: 患者として、マイページで現在の予約状況を確認したい。なぜなら、次回の予約日時をいつでも手元で確認したいから。（※キャンセル・変更は他ルールの「電話対応のみ」と統一し、マイページ上は確認のみとする）
- [x] **US-P15** [Must]: 患者として、マイページで自分の基本情報（連絡先・住所等）を変更したい。なぜなら、最新の情報を常に正しく保ちたいから。なお、氏名の変更は誤操作防止のため患者自身では不可とする。
- [x] **US-P12** [Must]: 患者として、予約を変更したい場合は電話で連絡したい。なぜなら、スタッフと直接やり取りして確実に変更したいから。

### スタッフとして

- [x] **US-S01** [Must]: スタッフとして、メールアドレスとパスワードでログインしたい。なぜなら、患者データに安全にアクセスする必要があるから。
- [x] **US-S02** [Must]: スタッフとして、ゲストログインで素早くアクセスしたい。なぜなら、パスワードなしで即座に管理画面を確認したい場面があるから。
- [x] **US-S03** [Must]: スタッフとして、本日の来院予約・来院者を一覧で確認したい。なぜなら、当日の患者対応を効率的に進める必要があるから。
- [x] **US-S04** [Must]: スタッフとして、電話予約を受けた際に予約を代理入力したい。なぜなら、HPを使わない患者の予約にも対応する必要があるから。
- [x] **US-S05** [Must]: スタッフとして、登録済み患者を検索・一覧表示したい。なぜなら、特定の患者情報に素早くアクセスしたいから。
- [x] **US-S06** [Must]: スタッフとして、施術後に店内で患者の次回予約を入力したい。なぜなら、患者がその場で次回日時を決められるから。
- [x] **US-S07** [Must]: スタッフとして、患者からの電話で予約変更を受け付けたい。なぜなら、予約変更は電話対応のみとし、スタッフが確実に処理する必要があるから。
- [x] **US-S08** [Must]: スタッフとして、患者の詳細情報と来院履歴を一画面で確認したい。なぜなら、前回の症状や経過を把握した上で施術したいから。
- [x] **US-S09** [Must]: スタッフとして、来院記録の詳細を確認・編集したい。なぜなら、患者入力の誤りをスタッフ側で修正する必要があるから。
- [x] **US-S10** [Must]: スタッフとして、患者の基本情報（氏名・連絡先・住所等）を更新したい。なぜなら、結婚による改姓や引越しに対応する必要があるから。
- [ ] **US-S11** [Could]: スタッフとして、入力された部位から推測されるリスクをAI分析で確認したい。なぜなら、施術の参考情報として活用したいから。
- [ ] **US-S12** [Should]: スタッフとして、患者情報（特に氏名などの重要情報）の変更履歴ログを確認したい。なぜなら、いつ・誰が・どう変更したかを追跡できるようにしておきたいから。

## 4. 機能要件

### A. 患者モード（タブレット入力画面）

#### A-1. ウェルカム画面 [Must]
- 清潔感のあるメインビジュアルを表示する
- 「受付を始める」ボタンを画面中央に配置する
- ボタンタップで基本情報入力画面に遷移する

#### A-2. 基本情報入力 [Must]
- 以下のフィールドを入力フォームとして表示する（shadcn/ui Formを使用）:
  - **氏名**（姓・名）: 必須、テキスト入力
  - **生年月日**: 必須、日付選択（デートピッカー）
  - **電話番号**: 必須、電話番号形式バリデーション付きテキスト入力
  - **住所**: 必須、テキスト入力
  - **メールアドレス**: 任意、メール形式バリデーション付きテキスト入力
- 入力完了後「次へ」ボタンで部位選択画面に遷移する
- **患者紐付け処理**: 氏名（姓＋名）と生年月日の組み合わせでprofilesテーブルを検索し、一致する既存患者がいれば該当患者の新規来院記録として登録する。一致しなければ新規患者として登録する。

#### A-3. 部位選択（ボディマップ） [Must]
- 人体のSVG画像を表示する
- 以下の主要部位をタップで選択可能にする: 頭、首、肩（左右）、背中（上部・下部）、腰、股関節（左右）、膝（左右）、足首（左右）
- 選択された部位はアクセントカラー（#FF4D4D）でハイライト表示する
- 複数部位の同時選択を可能にする
- 最低1箇所以上の選択を必須とする

#### A-4. 詳細入力 [Must]
- **症状の強さ**: 1〜10のスライダーで入力（デフォルト値: 5）
- **発症時期**: 以下の選択肢から1つ選択する（必須）:
  - 「本日」「2〜3日前」「1週間前」「2週間〜1ヶ月前」「1ヶ月以上前」
- **自由記述欄** [Should]: 最大500文字のテキストエリア（任意入力）
- 「送信」ボタンで入力内容をデータベースに保存する

#### A-5. 完了画面 [Must]
- 「ご入力ありがとうございました。スタッフをお呼びください。」というメッセージを表示する
- サクセスカラー（#28A745）でメッセージを装飾する
- 10秒後に自動的にウェルカム画面に戻る（次の患者のため）

### C. 予約機能（HP公開画面 + 管理者画面）

#### C-1. HP予約画面（患者向け・公開ページ） [Must]
- 院のHP（公開ページ）からアクセス可能な予約フォームを提供する
- 以下の情報を入力して予約する:
  - **氏名**（姓・名）: 必須
  - **電話番号**: 必須
  - **メールアドレス**: 必須（確認メール送信先）
  - **希望日時**: 必須、カレンダーから日付選択 + 時間帯選択（30分刻み）
  - **症状メモ**: 任意、簡単な症状の説明
- 予約完了後に画面上に確認メッセージを表示する
- 予約完了後に入力されたメールアドレスへ予約確認メールを送信する（予約日時・院の連絡先を記載）
- 営業時間外・定休日は選択不可とする
- 予約変更はHP上では不可。電話での連絡を案内する

#### C-2. 予約代理入力（管理者画面・電話予約対応） [Must]
- 管理者が電話予約を受けた際に、予約を代理で入力できる
- 既存患者の場合は患者検索から選択して予約作成する
- 新規患者の場合は氏名・電話番号を入力して予約作成する
- 希望日時・症状メモを入力する

#### C-3. 次回予約（店内予約） [Must]
- スタッフが施術後に、管理画面から患者の次回予約を入力できる
- 患者を選択（当日の来院者から選択、または患者検索から選択）して予約作成する
- 希望日時を入力する
- 予約経路は「店内」として記録する

#### C-4. 予約変更（電話対応のみ） [Must]
- 予約変更はHP上では受け付けない
- スタッフが管理画面から予約日時の変更・キャンセルを操作する
- HP予約画面には「予約の変更・キャンセルはお電話にてお願いいたします」と案内を表示する

#### C-5. 予約一覧・管理（管理者画面） [Must]
- 本日以降の予約を日別で一覧表示する
- 各予約には以下を表示する: 患者氏名、予約日時、症状メモ、予約経路（HP / 電話 / 店内）
- 予約の日時変更・キャンセル操作が可能
- ダッシュボードに本日の予約一覧を統合して表示する

### D. 患者マイページ（Web画面）[New]

#### D-1. 認証機能 [Must]
- 患者専用のログイン画面（メールアドレスとパスワードによる認証）
- 新規アカウント登録、および既存の患者情報との紐付け機能
- パスワードリセット機能

#### D-2. 予約・履歴管理 [Must]
- 現在の予約状況（日時・詳細）の確認
- ※予約のキャンセル・変更はマイページ上では不可とし、「お電話にてご連絡ください」と案内する（業務ルールの統一）。
- 次回予約の新規作成
- 過去の来院履歴の閲覧

#### D-3. 登録情報管理 [Must]
- 現在の登録情報（電話番号・住所・メールアドレス）の確認と編集
- ※氏名（姓名・旧姓）の変更機能は患者側には提供せず、「氏名の変更をご希望の場合はスタッフにお申し付けください」と案内のみ表示する。

### B. 管理者モード（PC/タブレット管理画面）

#### B-1. 認証 [Must]
- **ログイン画面**: メールアドレスとパスワードの入力フォームを表示する
- **ゲストログイン**: 「ゲストとしてログイン」ボタンを設置する。パスワード入力なしでSupabase匿名認証を利用してログインする。ゲストは通常の管理者と同等の権限を持つ。
- **ログアウト**: ヘッダーにログアウトボタンを配置する
- 未認証状態で管理者画面にアクセスした場合、ログイン画面にリダイレクトする

#### B-2. ダッシュボード [Must]
- 本日の予約一覧と来院者一覧を表示する
- **予約一覧**: 本日の予約を時間順で表示する。各予約には患者氏名、予約時間、予約経路（HP / 電話）、症状メモを表示する
- **来院者一覧**: 本日の来院記録を時系列で表示する（最新順）。各来院記録には患者氏名、来院時刻、選択部位（アイコンまたはバッジ）、痛みの強さを表示する
- 来院記録をクリックすると履歴詳細画面に遷移する

#### B-3. 患者一覧 [Must]
- 登録済み患者の一覧をテーブル形式で表示する
- データ増加によるパフォーマンス低下を防ぐため、1ページあたり50件のページネーション（または無限スクロール）とする
- デフォルトの表示順は最新の登録順（新しい患者が上）とする
- 氏名での検索機能を提供する
- 各行には氏名、生年月日、電話番号、最終来院日、初診/再診バッジを表示する
  - **初診**: 来院記録が0件の患者（予約のみで未来院）
  - **再診**: 来院記録が1件以上ある患者
- 患者名クリックで患者詳細画面に遷移する

#### B-4. 患者詳細 [Must]
- 選択した患者の基本情報（氏名、生年月日、電話番号、住所、メールアドレス）を表示する
- 患者基本情報の編集ができる（氏名・旧姓・電話番号・住所・メールアドレスの変更が可能）
- ※氏名変更などの重要情報の変更履歴（監査ログ）を残し、画面上で確認できるようにする
- 「次回予約を登録する」ボタンから予約登録フォームに遷移できる
- 来院履歴一覧を時系列（新しい順）で表示する。件数増加を考慮しページネーションを実装する。各来院記録には来院日時、選択部位（バッジ）、痛みの強さ、発症時期を表示する
- 来院記録の行をクリックすると来院記録詳細に遷移する

#### B-5. 来院記録詳細 [Must]
- 選択した来院記録の全内容を表示する: 来院日時、選択部位（ボディマップのハイライト表示）、痛みの強さ、発症時期、自由記述
- 来院記録の編集ができる（部位、痛みの強さ、発症時期、自由記述を変更可能）
- 来院記録の削除ができる（確認ダイアログを表示してから削除）

#### B-6. 予約登録・編集 [Must]
- 管理者画面から新規予約（電話予約・店内予約）を登録できる（C-2・C-3と共通フォーム）
- 既存予約の日時変更・キャンセルができる（C-4と共通フォーム）
- 患者詳細画面とダッシュボードの予約一覧から各操作に遷移できる

#### B-7. AI分析エリア [Won't — 将来拡張]
- 入力された部位から推測されるリスクを表示するカード
- MVP時点ではモック表示のみ（静的テキスト）とする

## 5. データベース設計（Supabase）

### profiles テーブル（患者情報）
| カラム名 | 型 | 制約 | 説明 |
|----------|------|------|------|
| id | uuid | PK, auto-gen | 患者ID |
| full_name | text | NOT NULL | 氏名（姓名結合） |
| previous_name | text | | 旧姓（任意） |
| birthday | date | NOT NULL | 生年月日 |
| phone | text | NOT NULL | 電話番号 |
| address | text | NOT NULL | 住所 |
| email | text | | メールアドレス（任意） |
| created_at | timestamptz | NOT NULL, default now() | 登録日時 |

- **紐付けキー**: `full_name` + `birthday` のユニーク制約で既存患者を識別する

### visits テーブル（来院記録）
| カラム名 | 型 | 制約 | 説明 |
|----------|------|------|------|
| id | uuid | PK, auto-gen | 来院記録ID |
| patient_id | uuid | FK → profiles.id, NOT NULL | 患者ID |
| pain_points | jsonb | NOT NULL | 選択部位の配列 (例: ["neck", "shoulder_left"]) |
| intensity | integer | NOT NULL, CHECK(1-10) | 痛みの強さ (1〜10) |
| onset | text | NOT NULL | 発症時期 ("today", "2_3_days", "1_week", "2w_1m", "over_1m") |
| memo | text | | 自由記述（任意） |
| created_at | timestamptz | NOT NULL, default now() | 来院日時 |

### appointments テーブル（予約）
| カラム名 | 型 | 制約 | 説明 |
|----------|------|------|------|
| id | uuid | PK, auto-gen | 予約ID |
| patient_id | uuid | FK → profiles.id | 既存患者の場合の紐付け（新規の場合はNULL） |
| patient_name | text | NOT NULL | 予約者氏名 |
| phone | text | NOT NULL | 予約者電話番号 |
| appointment_date | date | NOT NULL | 予約日 |
| appointment_time | time | NOT NULL | 予約時間 |
| symptom_memo | text | | 症状メモ（任意） |
| email | text | | メールアドレス（HP予約時は必須、電話・店内予約時は任意） |
| source | text | NOT NULL, CHECK('hp','phone','in_store') | 予約経路（HP / 電話 / 店内） |
| status | text | NOT NULL, default 'confirmed' | 予約状態（confirmed / cancelled） |
| created_at | timestamptz | NOT NULL, default now() | 予約登録日時 |

### staffs テーブル（管理者情報）
| カラム名 | 型 | 制約 | 説明 |
|----------|------|------|------|
| id | uuid | PK, FK → auth.users.id | スタッフID（Supabase Auth連携） |
| email | text | NOT NULL, UNIQUE | メールアドレス |
| display_name | text | NOT NULL | 表示名 |
| created_at | timestamptz | NOT NULL, default now() | 登録日時 |

### audit_logs テーブル（変更履歴ログ・追加検討）
| カラム名 | 型 | 制約 | 説明 |
|----------|------|------|------|
| id | uuid | PK, auto-gen | ログID |
| table_name | text | NOT NULL | 変更対象テーブル（例: `profiles`） |
| record_id | uuid | NOT NULL | 変更対象レコードID |
| action | text | NOT NULL | 操作内容（`UPDATE` 等） |
| old_data | jsonb | | 変更前データ |
| new_data | jsonb | | 変更後データ |
| changed_by | uuid | FK → staffs.id | 操作を行ったスタッフID |
| created_at | timestamptz | NOT NULL, default now() | 変更日時 |

### RLSポリシー方針
- **profiles**: 認証済みスタッフは全レコード読み取り可。患者モード（匿名）からはINSERT・UPDATEのみ許可。
- **visits**: 認証済みスタッフは全レコード読み取り可。患者モード（匿名）からはINSERTのみ許可。
- **appointments**: 認証済みスタッフは全レコード読み取り・書き込み可。HP予約（匿名）からはINSERTのみ許可。
- **staffs**: 本人のレコードのみ読み取り可。

## 6. 非機能要件

### パフォーマンス
- 患者モードの画面遷移は300ms以内に完了すること
- ダッシュボードの来院者一覧は1秒以内に表示すること
- 患者一覧の検索結果は500ms以内に返すこと

### セキュリティ
- 管理者画面はSupabase Authで認証を必須とする
- ※ポートフォリオの評価用としてゲストログイン機能を開放しているが、実運用環境（Production）を想定した場合、本機能は環境変数等で無効化する設計とする。
- 全テーブルでRow Level Security（RLS）を有効化する
- パスワードは8文字以上とする（Supabase Authのデフォルト設定に準拠）
- APIルートではZodによる入力バリデーションを実施する
- 患者モードのタブレットでは前の患者の入力内容が残らないようにする（完了画面からウェルカム画面への遷移時にフォーム状態をリセット）

### アクセシビリティ
- タブレット操作を前提とし、タッチターゲットは最低44x44pxを確保する
- フォントサイズは最小16px（患者モード）とする
- カラーコントラスト比はWCAG 2.1 AA基準（4.5:1以上）を満たす

### 可用性・エラーハンドリング
- Vercel + Supabaseのマネージドサービスに依存する（SLA準拠）
- API通信失敗時やオフライン時は、進行状況をブロックし「通信エラーが発生しました。スタッフにお声がけください」と画面に明示する。

## 8. 制約条件

### 技術的制約
- Supabaseの無料プランの制限内で運用する（データベース500MB、Auth 50,000 MAU）
- Vercelの無料プランの制限内で運用する
- タブレットのブラウザ（Safari / Chrome）で動作すること

### スコープ制約
- MVPスコープ: 患者モード全画面 + 患者マイページ全画面 + 予約機能（HP予約 + 電話予約代理入力） + 管理者モードのダッシュボード・患者一覧・履歴確認・予約管理
- 自動紐付け（名寄せ）の仕様について: 本MVPでは「氏名＋生年月日」の完全一致で患者を紐付ける。実運用では入力ブレによる二重登録が発生し得るため、将来拡張として管理者側での「名寄せ（データ統合）機能」を想定している。
- AI分析機能はMVPスコープ外（将来拡張として記録）
- 多言語対応はスコープ外（日本語のみ）
- 印刷機能はスコープ外

## 9. 用語集

| 用語 | 定義 |
|------|------|
| 患者モード | 来院患者がタブレットで症状を入力する画面群 |
| 管理者モード | スタッフがPC/タブレットで患者情報・来院記録を閲覧する画面群 |
| ボディマップ | 人体のSVG画像上で痛みの部位を選択するUIコンポーネント |
| 来院記録（Visit） | 患者が1回の来院時に入力した症状データ（部位・強度・発症時期・メモ）のセット |
| 患者紐付け | 氏名と生年月日の一致により、入力データを既存患者に関連付ける処理 |
| ゲストログイン | パスワード入力なしで管理者画面にアクセスする簡易認証方式（ポートフォリオ用のデモアクセス目的） |
| HP予約 | 患者が院のHP（公開ページ）からオンラインで行う来院予約。確認メールを送信する |
| 電話予約 | 患者が電話で予約し、スタッフが管理画面から代理入力する来院予約 |
| 店内予約（次回予約） | 施術後に店内でスタッフが患者の次回来院日を登録する予約 |
| 予約変更 | 予約日時の変更。電話対応のみで受け付け、HP上での変更は不可 |
| 初診 | 来院記録が0件の患者。予約のみで未来院の状態 |
| 再診 | 来院記録が1件以上ある患者 |
